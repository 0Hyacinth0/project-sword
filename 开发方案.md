设计并开发一款功能完备的回合制文字冒险网页游戏是一项激动人心但也极具挑战的工程。既然游戏涵盖了多人副本和PVP对战，我们需要**在"本地存储"和"防作弊（服务器权威）"之间取得现实的平衡**——纯本地存储极易被篡改，因此核心数据必须以云端（后端）为准，本地仅作缓存和表现层状态管理。

以下是为您量身定制的完整设计与开发架构方案，兼顾了技术实现的可行性与游戏体验的深度。

### 第一阶段：游戏核心系统设计

#### 1. 角色系统 (Character System)
角色是玩家交互的基石，我们需要建立一套严谨的数值转换模型。

##### 1.1 数据库表结构 (characters table)

| 字段 | 类型 | 说明 |
| :--- | :--- | :--- |
| `id` | UUID / Auto Increment | 唯一标识 |
| `user_id` | 外键 → users.id | 所属账号 |
| `name` | String (唯一索引) | 角色名 |
| `job_type` | Enum: WARRIOR / MAGE / HUNTER | 职业类型 |
| `level` | Integer (默认 1) | 等级 |
| `exp` | Integer (默认 0) | 经验值 |
| `base_str` | Integer | 基础力量 |
| `base_int` | Integer | 基础智力 |
| `base_agi` | Integer | 基础敏捷 |
| `created_at` | Timestamp | 创建时间 |

> **限制**：每个账号最多创建 **3** 个角色。

##### 1.2 职业与属性设定

| 职业 | 主属性 | 核心衍生属性倾向 | 战斗定位 |
| :--- | :--- | :--- | :--- |
| **战士** | 力量 (STR) | 1点力量 = 5 生命值 + 2 物理攻击力 | 高生存，稳定物理输出，吸收伤害 |
| **法师** | 智力 (INT) | 1点智力 = 5 魔法值 + 2 魔法攻击力 | 脆皮，高爆发，群体AOE伤害 |
| **猎人** | 敏捷 (AGI) | 1点敏捷 = 1 物理攻击 + 闪避/暴击率加成 | 高机动，高暴击，单体物理爆发 |

##### 1.3 职业配置表 (Static Config)

后端硬编码或数据库维护的职业静态数据，用于前端展示：

| 职业 | 力量 (STR) | 智力 (INT) | 敏捷 (AGI) | 生存 | 攻击 | 速度 |
| :--- | :---: | :---: | :---: | :---: | :---: | :---: |
| **战士 (Warrior)** | 10 | 3 | 5 | 5 | 3 | 2 |
| **法师 (Mage)** | 2 | 12 | 4 | 1 | 5 | 3 |
| **猎人 (Hunter)** | 5 | 4 | 11 | 2 | 4 | 5 |

> 雷达图维度：生存、攻击、速度，用于角色创建页的职业对比展示。

##### 1.4 加点机制
角色每次升级获得自由属性点。前端展示加点前后的面板数值预览，后端校验加点总数是否合法。

##### 1.5 角色创建相关接口

**GET `/api/characters`** — 获取账号下的角色列表

* 逻辑：根据当前登录的 `user_id` 查询所有角色。
* 返回：角色数组。如果数组长度为 0，前端自动跳转至创建页面。

**POST `/api/characters/create`** — 创建角色

* 请求参数：

```json
{
  "name": "剑圣无名",
  "job_type": "WARRIOR"
}
```

* 校验逻辑（核心）：
    1. 查询该 `user_id` 下的角色数量，若已达到 **3** 个，返回 `403` 错误（禁止创建）。
    2. 校验 `name` 是否已存在（全局唯一）。
    3. 校验 `job_type` 是否合法（WARRIOR / MAGE / HUNTER）。
* 处理：写入数据库，初始化 LV1 属性。

##### 1.6 前端页面设计

**角色选择页 (Character Selection)**

* 布局：采用 3 个并列的卡片槽位。
* 状态展示：
    * **已有角色**：展示角色名、等级、职业图标、最后在线时间。
    * **空槽位**：展示巨大的 "+" 号，引导点击创建。
* 进入游戏：点击已有角色卡片，将 `character_id` 存入全局状态（Pinia），进入主地图。

**角色创建页 (Character Creation)**

提升游戏"质感"的关键页面：

* **职业展示区（左侧/中间）：**
    * 雷达图实现：使用 ECharts 的 Radar Chart，动态根据选择的职业切换数据。
    * 示例技能：准备 1-2 个该职业的高级技能图标和描述。
* **职业交互区（右侧）：**
    * 职业切换按钮：战士/法师/猎人。切换时，主属性数值动态跳动变化。
    * ID 输入框：增加"随机取名"按钮（调用随机词库）。
* **响应式适配：**
    * 手机端雷达图缩小，技能介绍可采用滑动查看，确保不遮挡"开始冒险"按钮。

#### 2. 物品与装备系统 (Item & Equipment System)
文字游戏由于缺乏华丽的画面，装备驱动（Loot）是核心爽点。

* **背包模块：** 采用基于数组或字典的数据结构。支持标签分类（全部、消耗品、材料、装备）。
* **装备栏划分：** 武器 (Weapon)、头部 (Head)、胸部 (Chest)、腿部 (Legs)、饰品 (Accessory) x2。
* **装备品质体系：**

| 品质等级 | 颜色标识 | 基础属性成长 | 附加随机词条数 |
| :--- | :--- | :--- | :--- |
| 普通 (Normal) | 白色 | 100% | 0 |
| 稀有 (Rare) | 蓝色 | 120% | 1-2 |
| 史诗 (Epic) | 紫色 | 150% | 3-4 |
| 传说 (Legendary)| 橙色 | 200% | 4+ 及 特殊被动效果 |

* **套装机制 (Set Bonus)：** 装备数据结构中包含 `setId`。当玩家装备变动时，系统遍历全身装备，计算同 `setId` 的数量，达到 2件/4件/6件 时激活额外属性增益（Buff）。

#### 2.5 战宠系统 (Pet System)

战宠是角色的战斗伙伴，在副本中掉落获取。战宠作为独立战斗单位参与战斗，同时给主人提供额外属性加成，具备完整的养成深度。

##### 2.5.1 核心规则

| 规则项 | 设定 |
|:---|:---|
| 背包容量 | 每个角色最多拥有 **3** 只战宠 |
| 出战数量 | 同时只能 **1** 只出战 |
| 获取途径 | **副本掉落**（通关结算时有概率获得战宠蛋/契约） |
| 战斗定位 | 独立战斗单位 + 给主人提供属性加成 |
| 成长系统 | 升级、进化、技能学习、装备穿戴 |

##### 2.5.2 数据库表结构

**战宠类型表 (pet_types)** — 静态配置表

| 字段 | 类型 | 说明 |
| :--- | :--- | :--- |
| `pet_type_id` | Integer (PK) | 战宠类型 ID |
| `name` | String | 战宠名称 |
| `element` | Enum: 1-火, 2-水, 3-风, 4-地, 5-光, 6-暗 | 属性类型 |
| `rarity` | Enum: 1-N, 2-R, 3-SR, 4-SSR | 品质 |
| `base_hp` | Integer | 基础生命 |
| `base_attack` | Integer | 基础攻击 |
| `base_defense` | Integer | 基础防御 |
| `base_speed` | Integer | 基础速度 |
| `hp_growth` | Decimal | 每级生命成长 |
| `attack_growth` | Decimal | 每级攻击成长 |
| `defense_growth` | Decimal | 每级防御成长 |
| `speed_growth` | Decimal | 每级速度成长 |
| `evolve_to` | Integer (FK → pet_types) | 进化目标类型 ID，0 为最终形态 |
| `evolve_level` | Integer | 进化所需等级，0 为不可进化 |
| `icon` | String | 图标资源路径 |
| `description` | String | 背景描述 |

**角色战宠表 (character_pets)** — 玩家拥有的战宠实例

| 字段 | 类型 | 说明 |
| :--- | :--- | :--- |
| `id` | UUID (PK) | 战宠实例唯一标识 |
| `character_id` | UUID (FK → characters) | 所属角色 |
| `pet_type_id` | Integer (FK → pet_types) | 战宠类型 |
| `nickname` | String | 战宠昵称，默认取类型名 |
| `level` | Integer (默认 1) | 等级 |
| `experience` | Integer (默认 0) | 经验值 |
| `is_active` | Boolean (默认 false) | 是否出战 |
| `skill_slot_1` | Integer (FK → pet_skills, 可 null) | 技能槽位 1 |
| `skill_slot_2` | Integer (FK → pet_skills, 可 null) | 技能槽位 2 |
| `skill_slot_3` | Integer (FK → pet_skills, 可 null) | 技能槽位 3 |
| `equip_armor` | UUID (FK → character_items, 可 null) | 装备位：护甲 |
| `equip_accessory` | UUID (FK → character_items, 可 null) | 装备位：饰品 |
| `created_at` | Timestamp | 获得时间 |
| `updated_at` | Timestamp | 更新时间 |

> **限制**：每个角色最多 3 条记录，同一角色下 `is_active = true` 最多 1 条。

**战宠已学技能表 (pet_learned_skills)** — 多对多关联

| 字段 | 类型 | 说明 |
| :--- | :--- | :--- |
| `id` | UUID (PK) | 唯一标识 |
| `pet_id` | UUID (FK → character_pets) | 战宠实例 |
| `skill_id` | Integer (FK → pet_skills) | 已学技能 ID |

| `power` | Integer | 技能威力倍率（%） |
| `cooldown` | Integer | 冷却回合数 |

| `item_id` | Integer (FK → items) | 所需材料 ID |
| `quantity` | Integer | 所需数量 |

##### 2.5.3 属性计算公式

**战宠实际属性 = (基础属性 + 等级 × 成长值) × 品质倍率 + 装备加成**

```
hp = (base_hp + level × hp_growth) × rarity_multiplier + equip_hp
attack = (base_attack + level × attack_growth) × rarity_multiplier + equip_attack
defense = (base_defense + level × defense_growth) × rarity_multiplier + equip_defense
speed = (base_speed + level × speed_growth) × rarity_multiplier + equip_speed
```

**品质倍率对照：**

| 品质 | 倍率 |
|:---:|:---:|
| N | 1.0 |
| R | 1.2 |
| SR | 1.5 |
| SSR | 2.0 |

##### 2.5.4 主人属性加成

出战战宠会给主人提供额外属性加成，加成值基于战宠自身属性的一定百分比：

```
bonusHp = floor(pet.hp × 0.1 × rarity_multiplier)
bonusAttack = floor(pet.attack × 0.1 × rarity_multiplier)
bonusDefense = floor(pet.defense × 0.1 × rarity_multiplier)
```

> **说明**：品质越高，给主人的加成比例也越高，鼓励玩家追求高品质战宠。

##### 2.5.5 战宠战斗行为

- 战宠作为独立战斗单位参与回合制战斗，有独立的 HP 条和行动条
- 每回合根据速度值决定行动顺序（与角色、怪物共享行动条）
- 战宠 AI 自动使用已装备的技能，优先使用高威力技能（冷却中则顺延）
- 战宠死亡时，主人失去属性加成，但不会直接影响主人 HP
- 战斗结束后战宠 HP 自动恢复满

##### 2.5.6 进化规则

- 战宠达到指定等级 + 拥有足够的进化材料时可触发进化
- 进化后 `pet_type_id` 变更为进化形态，属性重新计算（保留 80% 当前经验进度）
- 进化可能习得进化专属技能（`learn_level = 0` 表示进化专属）
- 已装备的技能和装备在进化后保留

##### 2.5.7 副本掉落规则

| 副本类型 | 掉落概率 | 品质分布 |
|:---|:---:|:---|
| 普通副本 | 5% | N 80%, R 18%, SR 2% |
| 精英副本 | 15% | N 50%, R 35%, SR 13%, SSR 2% |
| 团队副本 Boss | 30% | R 40%, SR 45%, SSR 15% |

> **说明**：掉落的战宠类型由副本所在区域决定。同一副本可掉落多种不同类型的战宠。

##### 2.5.8 前端页面设计

**战宠主页面（Tab 页或独立页面）**

* **战宠列表区：** 3 个卡片槽位，展示战宠头像、昵称、等级、品质边框颜色。当前出战战宠有"出战"标签。空槽位显示 "+" 号引导文字。
* **战宠详情面板：** 点击战宠卡片后展开，包含：
    * 属性面板（HP/攻击/防御/速度），含装备加成分项展示
    * 技能槽位（3 个），点击可进入技能替换弹窗
    * 装备槽位（2 个：护甲 + 饰品），点击可进入装备选择弹窗
    * 进化按钮（满足条件时高亮，不满足时灰色且显示条件）
    * 喂食按钮（打开道具选择弹窗，选择经验道具和数量）
    * 重命名按钮

**战宠图鉴页面**

* 网格布局展示所有战宠类型，已拥有的正常显示，未拥有的显示为剪影 + "???"
* 点击进入图鉴详情：基础属性雷达图、进化链预览、技能列表
* 收集进度统计："已收集 12/50"

#### 3. 战斗与探索系统 (Combat & Exploration System)
* **回合制引擎：** 基于速度（或敏捷）决定出手顺序（行动条 Action Bar 机制）。状态机包含：`回合开始 -> 判断Buff/Debuff -> 行动选择 (攻击/技能/物品) -> 伤害计算与闪避/暴击判定 -> 结算 -> 回合结束`。
* **野外刷怪 (PVE)：** 按照区域划分（如：迷雾森林 Lv.1-10，白骨荒野 Lv.11-20）。支持挂机离线收益（根据玩家战斗力计算离线每小时收益率）。
* **副本系统 (Dungeons)：** * *单人副本：* 消耗体力值，产出特定材料。
    * *多人副本：* 基于 WebSocket 的实时大厅，房主开启挑战。怪物血量动态根据人数缩放。

#### 4. 社交与竞技系统 (Social & PvP System)
* **账号体系：** JWT (JSON Web Tokens) 认证，包含注册、登录、找回密码。
* **好友与组队：** 互相关注形成好友关系。支持在线状态广播，一键发送组队邀请链接或弹窗。
* **PVP 竞技场：** * *非实时竞技：* 挑战离线玩家镜像（AI自动反击），适合前期。
    * *实时匹配：* 采用 Elo 积分算法，使用 WebSocket 进行双端状态同步，具有 30 秒出招倒计时。

### 第二阶段：技术选型与架构 (Technical Architecture)

为了保证"响应式"、"数据安全"和"高并发"，推荐以下现代 Web 技术栈：

* **前端 (Frontend):**
    * **框架:** Vue 3 + TypeScript。TypeScript 对于RPG复杂的对象数据结构（物品、属性）有极好的类型保护。
    * **样式:** 原生 CSS + CSS 变量，遵循 DESIGN.md Liquid Glass 设计规范。
    * **状态管理:** Pinia 管理玩家当前状态和背包。
    * **图表:** ECharts（角色创建页雷达图）。
* **后端 (Backend):**
    * **框架:** JeecgBoot（Spring Boot）。
    * **实时通信:** Socket.io 或原生 WebSocket (用于多人副本和PVP)。
* **数据库 (Database):**
    * **持久层:** PostgreSQL / MySQL (存储用户资产、背包详情、装备属性表)。
    * **缓存与排行:** Redis (处理排行榜高速读写、PVP匹配队列、用户Session状态)。
* **数据安全与同步策略：** **务必采用服务器权威 (Server-Authoritative) 架构。** * 前端只负责发送操作指令（如"使用火球术攻击哥布林"）和播放动画效果。
    * 后端执行核心逻辑计算（命中率、暴击、扣血），并将结果回传前端展示。本地 `localStorage` 仅用于缓存用户设置和静态数据（如物品图标映射）。
